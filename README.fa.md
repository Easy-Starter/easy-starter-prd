## این مخزن کد چی هست؟

یک کیت شروع ساده، کاربردی و دو‌زبانه برای تبدیل ایده‌های محصولی به PRD، دیزاین بریف، اسپک، پلن و تسک‌های آماده برای توسعه.

قبل از اینکه حتی یک خط کد نوشته شود، این مخزن کد کمک می‌کند تیم به مهم‌ترین سوال‌های محصول جواب بدهد:

- **دقیقاً چه چیزی می‌سازیم؟**
- **برای چه کسی می‌سازیم؟**
- **چرا این محصول مهم است؟**
- **موفقیت را چطور اندازه می‌گیریم؟**
- **طراح، توسعه‌دهنده، QA و کدینگ ایجنت باید به چه سندی اعتماد کنند؟**

هدف ساده است: **مستندات منبع حقیقت تیم باشند؛ نه چت‌های پراکنده، حافظه‌ی افراد یا فرضیات پنهان.**

---

## چرا این مخزن کد ساخته شده؟

خیلی از پروژه‌ها به این دلیل شکست نمی‌خورند که تیم بلد نیست کد بزند.

معمولاً مشکل از اینجا شروع می‌شود که تیم قبل از شفاف شدن مسئله، کاربر، محدوده، edge caseها و معیار موفقیت وارد پیاده‌سازی می‌شود.

**Easy Starter PRD** یک مسیر آرام، مرحله‌به‌مرحله و قابل تکرار می‌دهد تا ایده خام به یک مشخصات محصولی قابل طراحی، توسعه، تست و لانچ تبدیل شود. این قالب برای فریلنسرها، تیم‌های کوچک، استارتاپ‌ها، محصول‌سازها و پروژه‌های AI-first مناسب است.

## جریان کاری پیشنهادی

هر مرحله کمک می‌کند ابهام کمتر شود و تصمیم‌ها برای محصول، طراحی، توسعه، تست و رشد شفاف‌تر باشند.

```text
ایده
  ↓
خلاصه یک‌صفحه‌ای محصول
  ↓
تحقیق
  ↓
PRD
  ↓
دیزاین بریف
  ↓
مشخصات UX/UI
  ↓
بریف فنی
  ↓
Feature Spec
  ↓
Plan
  ↓
تسک‌های تست‌محور
  ↓
پیاده‌سازی
  ↓
ریلیز
  ↓
یادگیری بعد از لانچ
```

## ساختار مخزن کد

```text
templates/                  قالب‌های انگلیسی محصول، طراحی و مهندسی
specs/_feature-template/       قالب قابل استفاده برای توسعه spec-driven
specs/features/                اسپک فیچرهای واقعی اینجا قرار می‌گیرند
prompts/                    پرامپت‌های انگلیسی برای محصول، طراحی و کدینگ ایجنت
checklists/                    گیت‌های بازبینی قبل از رفتن به مرحله بعد
examples/                      نمونه‌های تکمیل‌شده برای درک سریع‌تر
.github/                       قالب Issue و Pull Request
```

## از کجا شروع کنم؟

از این فایل شروع کن:

```text
templates/fa/00-product-one-pager.fa.md
```

یا اگر نسخه انگلیسی را ترجیح می‌دهی:

```text
templates/en/00-product-one-pager.md
```

بعد مرحله‌به‌مرحله جلو برو. بهتر است خیلی زود وارد پیاده‌سازی نشوی؛ چند دقیقه شفاف‌سازی در مستندات می‌تواند چندین ساعت بازکاری را کم کند.

## شناسه‌گذاری نیازمندی‌ها

از شناسه‌های ثابت در PRD، اسپک، تسک‌ها، کامیت‌ها، Pull Requestها و تست‌ها استفاده کن.

| شناسه      | معنی                         |
| ---------- | ---------------------------- |
| `GOAL-001` | هدف محصول یا بیزینس          |
| `USER-001` | سگمنت کاربر، پرسونا یا actor |
| `REQ-001`  | نیازمندی فانکشنال            |
| `NFR-001`  | نیازمندی غیرفانکشنال         |
| `AC-001`   | معیار پذیرش                  |
| `EVT-001`  | ایونت آنالیتیکس              |
| `RISK-001` | ریسک، فرضیه یا سوال باز      |
| `DEC-001`  | تصمیم محصولی، طراحی یا فنی   |

این کار باعث می‌شود هر تصمیم پیاده‌سازی به هدف اصلی محصول قابل ردیابی باشد.

## چجوری از این مخزن کد استفاده کنم؟

1. مخزن کد را clone کن یا به‌عنوان template استفاده کن.
2. اول Product One-Pager را کامل کن.
3. از پرامپت‌های داخل `prompts/` برای شفاف‌سازی بخش‌های مبهم کمک بگیر.
4. PRD و Design Brief را بنویس.
5. برای هر فیچر یک پوشه جدید در مسیر `specs/features/001-feature-name/` بساز.
6. فایل‌های داخل `specs/_feature-template/` را داخل پوشه فیچر کپی کن.
7. از روی Feature Spec فایل‌های `plan.md` و `tasks.md` را بساز.
8. از کدینگ ایجنت بخواه فقط طبق spec و taskها پیاده‌سازی کند.
9. در هر Pull Request شناسه‌های مرتبط با نیازمندی‌ها را ذکر کن.
10. بعد از ریلیز، یادگیری‌ها را مستند کن و source of truth را به‌روز نگه دار.

## کار با کدینگ ایجنت‌ها

این مخزن کد طوری طراحی شده که با کدینگ ایجنت‌ها، AI IDEها و فرایندهای spec-driven خوب کار کند.

یک قانون ساده:

> از ایجنت نخواه «کل اپلیکیشن را بسازد».  
> از ایجنت بخواه یک feature spec مشخص را با نیازمندی‌ها، معیار پذیرش، تست‌ها و محدودیت‌های واضح پیاده‌سازی کند.

همین تغییر کوچک باعث می‌شود توسعه با AI قابل پیش‌بینی‌تر، قابل نگهداری‌تر و قابل بازبینی‌تر باشد.
